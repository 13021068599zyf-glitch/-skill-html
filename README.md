# 用户研究综合分析工具
**User Research Synthesis Skill · 《智能设计系统与管理策略》课程项目**

将定性用户研究数据（访谈记录、问卷、工单）转化为结构化洞察报告的 AI 辅助工具。

---

## 项目简介

本工具通过 **8 步流水线 + 3 个人工审核节点（HITL）**，协助研究员完成从原始访谈文本到可交付洞察报告的全流程分析。调用 [Kimi（月之暗面）](https://platform.moonshot.cn) `moonshot-v1-32k` 模型，单文件 HTML，无需安装任何依赖，浏览器直接打开运行。

```
数据输入 → 预览确认 → 🤖 自动编码归组 → 🧑 主题审核
        → 🤖 命名标记 → 🧑 矛盾证据处置 → 🤖 洞察提炼
        → 🧑 洞察审核 → 🤖 生成报告
```

---

## 快速开始

### 1. 获取 Kimi API Key

前往 [platform.moonshot.cn/console/api-keys](https://platform.moonshot.cn/console/api-keys) 注册并创建 API Key。

### 2. 打开工具

下载 `user-research-skill.html`，用浏览器（推荐 Chrome / Edge）直接打开：

```
双击 user-research-skill.html
```

无需启动服务器，无需安装 Node.js 或 Python。

### 3. 填写信息并开始分析

1. 粘贴 Kimi API Key（仅保存在浏览器内存，不上传）
2. 填写研究问题和分析边界
3. 上传访谈文件（.txt / .csv / .docx / .xlsx）或直接粘贴文本
4. 点击「开始分析」

---

## 输入格式

### 文件上传（推荐）

支持格式：`.txt` · `.csv` · `.docx` · `.doc` · `.xlsx` · `.xls`

> ⚠️ `.docx` 和 `.xlsx` 解析需要加载外部库（mammoth.js / xlsx.js），需要网络连接。
> 离线使用请将文件转存为 `.txt` 或 `.csv`。

### 粘贴文本

多份访谈用 `===` 分隔：

```
=== 受访者 P001（健身博主，26岁） ===
访谈内容……

===

=== 受访者 P002（骑行爱好者，31岁） ===
访谈内容……
```

上传文件与粘贴文本可同时使用，内容会自动合并。

## 分析流程说明

| 步骤 | 类型 | 说明 |
|------|------|------|
| 数据预览确认 | 人工 | 确认文件提取内容完整，无乱码 |
| 步骤 1–4：编码归组 | 🤖 AI | 切分片段、初始编码、情感标注、编码归组 |
| **HITL-1：主题审核** | 🧑 人工 | 重命名、合并、删除分组；支持拖拽片段调整归属 |
| 步骤 5：命名标记 | 🤖 AI | 生成主题名称，标记矛盾证据 |
| **HITL-2：矛盾处置** | 🧑 人工 | 逐条决定：忽略噪声 / 作为独立发现 / 降低置信度 |
| 步骤 6：洞察提炼 | 🤖 AI | 生成洞察陈述、置信度、引语、行动建议 |
| **HITL-3：洞察审核** | 🧑 人工 | 批准 / 编辑 / 删除；可展开查看原始片段 |
| 步骤 8：生成报告 | 🤖 AI | 输出 Markdown 研究综合报告 |

## Skill 使用指南

`SKILL.md` 是本项目的核心操作文档，定义了 AI 在每个步骤应该做什么、怎么做、输出什么格式。以下是关键使用说明。

### 触发条件

当用户提供以下任意一种数据时，本 Skill 被激活：

- 访谈记录（.txt / .md / .docx）
- 问卷开放题导出（.csv / .xlsx）
- 客服工单 / 用户反馈日志（.csv / .json）

**推荐数据规模：** 5–30 份访谈记录，每份 500–3000 字。

---

### 研究问题与分析边界（必填）

启动前必须明确两个前置输入，否则 AI 会做泛化总结而非针对性分析：

| 字段 | 说明 | 示例 |
|------|------|------|
| **研究问题** | 本次研究要回答的核心问题 | `用户在首次使用产品时遇到的主要障碍是什么？` |
| **分析边界** | 哪些内容在范围内，哪些不分析 | `聚焦注册到完成首次任务的全流程体验，不含付费转化` |

---

### 三个 HITL 审核节点操作说明

#### HITL-1：主题聚类审核

AI 完成初始编码后，展示编码分组卡片。

**可执行的操作：**

| 操作 | 方式 |
|------|------|
| 重命名 | 直接点击分组标题，输入新名称 |
| 合并 | 点击「🔗 合并」，再点击目标分组卡片 |
| 删除 | 点击「✕」，该分组内片段不参与后续分析 |
| 拖拽调整 | 抓住片段左侧「⠿」手柄，拖入另一分组 |
| 新建分组 | 点击「+ 新增空分组」，再将片段拖入 |
| 折叠列表 | 点击「▼ 折叠片段」收起长列表 |

> **判断标准：** 两个分组语义相近但业务上不同类时保持独立；同类但命名不同时合并重命名。

#### HITL-2：矛盾证据处置

AI 标记出与主流主题相悖的片段，每条须选择处置方式：

| 选项 | 适用场景 | 对洞察的影响 |
|------|---------|------------|
| 忽略噪声 | 明显偏离主题，属于无效信息 | 不影响置信度 |
| 作为独立发现 | 少数但有价值的反向声音 | 在报告中单独列出 |
| 降低置信度 | 说明该主题存在分歧，需谨慎 | 洞察置信度下调 10–15% |

#### HITL-3：洞察质量审核

AI 生成洞察卡片后，逐条操作：

| 操作 | 说明 |
|------|------|
| ✓ 批准 | 洞察纳入最终报告 |
| ✏ 编辑 | 直接修改措辞，保存后自动标记为「已批准」 |
| ✕ 删除 | 排除低质量洞察（可点「↩ 恢复」撤销） |
| ▼ 查看原始片段 | 展开该主题下所有访谈原文，确认证据是否支撑洞察 |

> **质量判断标准：** 好的洞察是 15–30 字的可执行陈述，能直接回答"所以产品应该怎么做"，而不是现象描述（如"用户反映体验不好"）。

---

### 置信度说明

| 置信度 | 含义 | 建议操作 |
|--------|------|---------|
| ≥ 85% | 证据充分（5 条以上），结论可靠 | 直接批准 |
| 70–84% | 证据中等（3–4 条），基本可信 | 核查原始片段后批准 |
| 60–69% | 证据较少（2 条），需谨慎 | 建议编辑措辞或降级处理 |
| < 60% | 证据不足或有矛盾 | 建议删除或标注为初步发现 |

---

### 报告输出与使用

分析完成后生成 Markdown 格式报告，包含执行摘要、核心主题、洞察列表和推荐行动。

| 按钮 | 用途 |
|------|------|
| 📋 复制 Markdown | 粘贴到 Notion / 飞书文档 |
| ⬇ 下载报告 | 保存为 .md 文件 |
| 🔄 重新分析 | 清空状态从头开始（数据不保留） |

> ⚠️ 全程不要刷新页面，分析状态不会自动保存。

### 常见问题

**Q：分析卡住不动了怎么办？**
展开底部「▶ 运行日志」面板，查看最后一条日志。ERR 红色条目说明 API 调用失败，常见原因是 API Key 余额不足或输入文本过长。

**Q：JSON 解析失败怎么办？**
日志显示截断时工具会自动尝试修复。若修复失败，减少输入量（建议每次不超过 15 份记录）后重试。

**Q：主题分类结果不理想？**
在 HITL-1 大胆调整——合并过细的分组、删除无意义的兜底组、将错误归类的片段拖到正确分组。HITL-1 的调整质量直接决定后续所有输出的质量。

**Q：洞察建议太泛化怎么办？**
在 HITL-3 中直接编辑措辞，加入具体的目标用户、触发场景和预期影响。例如将"优化拍摄体验"改为"针对独自骑行用户，提供语音触发录制功能，解决戴手套无法操作的问题"。

## 隐私说明

- **API Key**：仅保存在浏览器内存，页面关闭后自动清除，不发送至任何第三方服务器
- **访谈文本**：会通过 HTTPS 发送至 Kimi（月之暗面）模型服务进行分析
- **使用前请完成脱敏**：将受访者姓名替换为编号（如 P001），删除手机号、地址等可识别信息

## 技术栈

- **前端**：纯 HTML / CSS / JavaScript，无框架
- **AI 模型**：Kimi `moonshot-v1-32k`（OpenAI 兼容接口）
- **文件解析**：mammoth.js（docx）、SheetJS xlsx.js（xlsx）
- **运行环境**：浏览器，无需服务器
