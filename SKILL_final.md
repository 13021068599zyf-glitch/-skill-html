# 用户研究综合分析技能

## 概述

本技能通过 **8 步流水线 + 3 个 HITL 审核节点**，将原始访谈文本转化为可执行洞察报告。调用 Kimi `moonshot-v1-32k` 模型，全程在浏览器内运行，无需安装任何依赖。

```
输入数据 → [预览确认] → 🤖编码归组 → 🧑主题审核 → 🤖命名标记
         → 🧑矛盾处置 → 🤖洞察提炼 → 🧑洞察审核 → 🤖生成报告
```

---

## 输入规格

| 维度 | 说明 |
|------|------|
| **文件格式** | `.txt` `.csv` `.docx` `.doc` `.xlsx` `.xls` |
| **输入方式** | 文件上传（支持多文件、拖拽）或直接粘贴文本 |
| **数据规模** | 5–30 份访谈记录，建议每份 500–3000 字 |
| **分隔方式** | 多份文本用 `===` 分隔；多文件自动按文件名分隔 |
| **前置要求** | 完成基础脱敏（姓名替换为 P001 等编号） |

---

## 执行流程

### 预览确认（数据载入后）

解析所有输入文件，展示每个来源的字符数、估算页数和内容预览（前 800 字符）。状态指示：
- 🟢 内容正常（≥500 字符）
- 🟡 内容较短（200–500 字符）
- 🔴 内容偏少（＜200 字符）

用户确认内容完整后，进入 AI 分析。

---

### 步骤 1–4：自动编码与归组（API 调用 #1）

**Prompt 核心指令：**
- 将访谈内容切分为有意义的片段（每段 50–150 字）
- 为每段打 2–4 词描述性编码（保留原始语言风格，不做解读）
- 标注情感倾向：`negative` / `positive` / `neutral`
- 标注情绪强度：`高` / `中` / `低`
- 将相似编码归为 5–7 个组

**输出 JSON 结构：**
```json
{
  "segments": [
    {
      "id": "S001",
      "participant": "P001",
      "text": "核心内容摘要",
      "code": "编码标签",
      "sentiment": "negative",
      "intensity": "高"
    }
  ],
  "codeGroups": [
    {
      "id": "G01",
      "tentativeName": "分组名称",
      "segmentIds": ["S001", "S002"],
      "codes": ["编码1", "编码2", "编码3"]
    }
  ]
}
```

**动态 max_tokens：** 根据输入长度自动计算，公式为 `min(16000, max(6000, ceil(inputLen/250) × 120 + 1000))`，避免截断。

**截断自动修复：** 若输出末尾不是 `}` 或 `]`，自动从末尾向前查找最后一个合法 JSON 边界进行修复。

---

### HITL-1：主题聚类审核

展示 AI 生成的编码分组卡片，每张卡片显示分组名称、片段数、代表性编码标签。

用户可执行：
- **重命名**：直接点击标题编辑
- **合并**：点击"合并"后选择目标分组
- **删除**：移除整个分组（其中片段不参与后续分析）
- **新增**：创建空分组

确认后继续。

---

### 步骤 5：主题命名与矛盾标记（API 调用 #2）

基于用户确认后的分组，模型执行：
1. 为每个分组生成 4–8 字主题名称 + 15–30 字描述
2. 识别与主流主题相矛盾的片段，说明矛盾原因

**输出 JSON 结构：**
```json
{
  "themes": [
    {
      "id": "T01",
      "groupId": "G01",
      "name": "主题名称",
      "subtitle": "核心张力描述"
    }
  ],
  "contradictions": [
    {
      "segmentId": "S005",
      "themeId": "T01",
      "contradictingNote": "矛盾原因说明"
    }
  ]
}
```

max_tokens：4000

---

### HITL-2：矛盾证据处置

展示每条矛盾片段（含原文、受访者、编码、情感），用户通过下拉菜单选择处置方式：

| 选项 | 含义 | 对下游影响 |
|------|------|-----------|
| 忽略噪声 | 视为无效数据 | 不影响置信度 |
| 作为独立发现 | 单独列出 | 在报告中独立呈现 |
| 降低主题置信度 | 证明存在分歧 | 该主题洞察置信度下调 10–15% |

---

### 步骤 6：洞察提炼（API 调用 #3）

基于确认的主题和矛盾处置决定，为每个主题提炼 1–2 条洞察。

**Prompt 核心要求：**
- 洞察陈述：15–30 字，清晰可执行，不写套话
- 置信度：依据证据条数 + 矛盾处置结果计算
- 支撑引语：2–3 条原文节选
- 行动建议：2 条具体建议

**置信度计算规则：**

| 证据条数 | 基础置信度 |
|---------|-----------|
| ≥ 5 条 | 85% 以上 |
| 3–4 条 | 70–80% |
| 2 条 | 60–70% |

有矛盾证据被标记为"降低置信度"时，额外下调 10–15%。

**输出 JSON 结构：**
```json
{
  "insights": [
    {
      "id": "I01",
      "themeId": "T01",
      "statement": "洞察陈述",
      "confidence": 85,
      "evidenceCount": 5,
      "quotes": ["引语1", "引语2"],
      "recommendations": ["建议1", "建议2"]
    }
  ]
}
```

max_tokens：5000

---

### HITL-3：洞察质量审核

展示每条洞察卡片，包含：
- 所属主题标签 + 审核状态
- 洞察陈述（可内联编辑）
- 置信度进度条 + 证据条数
- AI 提炼的支撑引语
- 行动建议标签
- **「查看原始片段」折叠面板**：点击展开该主题下的所有原始访谈片段，每条显示完整原文、受访者编号、编码标签、情感倾向、情绪强度

用户操作：
- **✓ 批准**：纳入报告
- **✏ 编辑**：内联修改措辞后保存
- **✕ 删除**：排除出报告
- **↩ 恢复**：撤销删除

---

### 步骤 8：生成报告（API 调用 #4）

基于用户批准的洞察，生成 Markdown 格式研究综合报告。

**报告结构：**
```markdown
# 用户研究综合报告

## 执行摘要（3–4 句）

## 研究方法
- 数据来源、受访者数量、分析边界

## 核心主题（每个主题 2–3 段）

## 顶级洞察（附置信度与行动建议）

## 推荐行动（按优先级排序）

## 附录
- 编码体系 / 受访者索引 / 矛盾证据处置记录
```

max_tokens：6000

报告支持页内渲染预览、一键复制 Markdown、下载为 `.md` 文件。

## 运行日志

工具底部常驻日志面板，记录每次 API 调用的：
- 模型名称、max_tokens、输入字符数
- 响应耗时、输出字符数、prompt/completion token 用量
- 三个 HITL 节点的用户操作详情
- 错误信息及自动修复结果

支持复制完整日志文本用于调试记录。

---

## 注意事项

- **HITL 节点不可跳过**：三个审核节点是质量保障机制，确保 AI 偏差在进入下一步前被人工修正
- **研究边界前置**：启动前必须明确填写研究问题和分析边界，否则 AI 无法确定主题范围
- **隐私保护**：受访者统一用编号，不使用真实姓名
- **引语保留原话**：编码和引语尽可能逐字保留受访者原始表达，不改写
- **矛盾证据不忽视**：AI 标记的矛盾证据须经人工处置，不得自动跳过
- **文件大小建议**：单次输入超过 30 份访谈时建议分批处理，避免 token 超限
